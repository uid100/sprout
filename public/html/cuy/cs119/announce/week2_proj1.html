Getting Started

As I mentioned, programming classes usually start out with very simple exercises to bridge the human-machine interface as quickly as possible. In other words, students provide simple instructions to get the computer to do something. If you continue with studying other programming languages, you'll find that this is usually a special program called "HelloWorld." This is true in almost every language I've ever seen, with minor variations and few exceptions. A quick search will reveal many of these.

In addition to simply getting started, another theme that I hope you saw was the realization that almost every program you'll ever write or use follows the same pattern:

  • it takes some sort of input,
  • does some amount of processing,
  • and responds with some output.

That can mean prompting the user for two numbers, multiplying them together, and displaying the result, or it can be waiting for a pedestrian to press a button, monitoring the traffic status, and cycling a traffic light to signal cross traffic to slow, then stop, and then lighting a Walk signal and starting a timer to trigger the next cycle. Think about any computer application that you use. Can you identify the inputs and the outputs?

This week you did this. You wrote usable and useful computer programs. You taught the computer how to respond to inputs and deliver a predictable result. That's quite an accomplishment. This class and your experiences will get more complex, but we'll build on this one step at a time.

 

Pseudocode

What is it?

In one of my advanced programming classes, I bring decks of playing cards to class and deal them to my students and I ask them to sort that set of cards in their hand. This is something that is fundamental to anyone who has ever played just about any card game. Next I ask them to each write down a set of instructions of how to sort a hand of cards and then trade those instructions with each other and try and follow the steps, explicitly. Every single step on the instructions needs to be followed, exactly as written. Yes, there are assumptions and questions and rules. What does it mean to "sort" them? Why is a 'Q' higher than a 'J'? Does sort mean left-to-right? .. low-to-high? .. is the 'Ace' the highest card or the lowest? What about suits (hearts, diamonds...)? Oh, and I have 'Jokers' in my card decks because my family uses them for some games. Some of the students will find them in their hand and choose how to deal with them in their instructions, but some might find them in their hand and not have any instructions written for them by another student who didn't think that they'd be there.

This brings up very real situations and common problems we run into when writing code. There are circumstances that might be encountered that the developer needs to consider. We'd like to think that the user will enter values the way we expect them to, but most errors in code exist because a user enters something that we didn't imagine. An application might ask for a user's birthday. Did the developer check to see that 31-April is not a valid date, and that although it's uncommon to be born on 29-Feb, it is not uncommon for babies to be born on 29-Feb.

But more importantly, is that sequence of instructions clear enough to follow? Is it complete and detailed? Is the sequence correct and not ambiguous? In other words is there one and only one way to follow them and get the correct result. If these things are all done correctly, then anyone you give those instructions will be able to follow them and get the same result, every time.

What are all of the steps to do something? What is the correct order of all of those steps?

Pseudocode is that set of clear instructions that you can give to anyone and they could follow them to get the same result.

What isn't it?

Pseudocode is not written in any programming language. In fact, the logic should be simple enough that you could give these instructions to someone who has no programming knowledge at all and follow them. The pseudocode also should make no assumptions of what language will be used to write the solution.

This sequence of logical instructions, which is commonly called an algorithm could be read by any programmer and they might decide which programming language best fits the solution, but that choice of language would make no difference to the algorithm that was written in pseudocode.

For our labs, do not write the solution in Python and then copy your code into a text file and call it 'pseudocode.' If everything in your set of simple, logical, sequenced instructions aren't completely clear to your roommate, or your great aunt Helen, or your son's basketball coach, then you might be missing something.

Keep it simple, logical, correct.

Do not submit a copy of your code and call it pseudocode.

 

Shebang!

Yes, that's really what it's called.

The '#' symbol has been called many things over the years, but most recently, in the computer world at least, it's been referred to as a 'hash' symbol. Instagram uses them to identify keywords. They call these 'hashtags' which is a tag (identifier) set off by a hash symbol.

The exclamation point ('!'), used in written languages, is used for other purposes in programming and is often pronounced as 'bang.'

Together the hash-bang has become known as a 'shebang' (pronounced: shuh-BANG) and is used in scripting or interpreted languages. This line needs to be at the very beginning of a file. The hash (#) tells the interpreter to ignore or do nothing with this line, but it tells most operating systems which interpreter to use. In the case of Python3, the common usage is "#!/usr/bin/env python3"

 

(Examples)

"#!/usr/bin/env bash" tells the shell reading this file to use the Bash shell to execute the file.

"#!/usr/bin/env pwsh" is sometimes used with Windows Powershell scripts.

 

Writing good code

Self-documenting code

Yes, good code needs to work, but like pseudocode, it should be clear and simple to follow. It might be somebody else that has to update or fix or reuse something that you've written, but just as commonly, it could be you (or me!) that has to open a program that we've written months or years before and we have to sort through it and try and remember what we were thinking at the time.

Good code starts with good identifiers. Give your variables logical names. A variable called "height" or "height_in_inches" or "centimeters" or even "cm" is pretty clear what kind of data it might contain. If your variables have clear, logical names, then the purpose of a statement such as
area_of_circle = value_of_pi * Radius * Radius
would be very clear, and good clear code is less likely to have errors.

Using Comments

Self-documented code is always a good idea, but sometimes, even with good name choices for variables and functions (or methods), comments are still helpful to keep the developers intent clear.

tax_rate = 0.0825 # local tax rate for El Cajon as of 1-Aug-2021

This line not only defines the value in one place, making it easy to update and use, but it also clarifies where that value originated.

X = 4 * sqrt(1-(a*a))
is also a very logical expression and one that might be in common use in statistical analysis, but without comments and better descriptions, its purpose is not clear to anyone.
